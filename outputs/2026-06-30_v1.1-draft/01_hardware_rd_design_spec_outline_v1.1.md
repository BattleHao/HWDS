# 硬件研发设计规范 V1.1 修正版大纲

| 项目 | 内容 |
|---|---|
| 文档名称 | 硬件研发设计规范 |
| 版本 | V1.1 草案 |
| 适用企业 | 江苏苏威尔科技有限公司 |
| 适用范围 | 教育装备类硬件产品研发、验证、导入、量产维护与工程变更 |
| 流程框架 | IPD 六阶段 + 项目分级裁剪 + 分阶段导入 |
| 文档定位 | 企业制度文件、项目执行指南、评审门禁依据、模板索引 |
| 修订重点 | 降低一次性导入难度，增强小团队可执行性和项目适配性 |

## 1. 总则

### 1.1 目的

建立统一、可执行、可评审、可追溯、可裁剪的硬件研发设计规范，解决当前研发流程不完整、执行偏弱、评审标准不统一、文档缺失、问题闭环不足和跨部门协同不稳定等问题。

### 1.2 建设目标

本规范不是一次性引入复杂流程，而是通过分阶段导入，逐步形成适合企业现状的硬件研发管理体系。

| 阶段 | 建设目标 | 重点动作 |
|---|---|---|
| 试运行期 | 先让关键项目跑起来 | 选 1-2 个项目试点，执行最小必需流程 |
| 固化期 | 让流程稳定执行 | 固化阶段评审、问题闭环、归档和变更机制 |
| 优化期 | 提升效率和质量 | 建立度量指标、裁剪规则、复盘改进机制 |

### 1.3 适用范围

适用于公司教育装备类产品相关硬件研发活动，包括：

- 全新产品开发。
- 平台型产品开发。
- 派生产品开发。
- 客户定制项目。
- 降本优化项目。
- 器件替代项目。
- 质量改进项目。
- 预研项目。

### 1.4 适用对象

适用于产品、项目、系统、硬件、PCB、固件、结构、测试、工艺、生产、采购、供应链、质量、售后及 IPD / PDT 相关人员。

### 1.5 基本原则

- 需求先行：需求未明确时，不得冻结详细设计。
- 方案先行：总体方案未评审时，不得正式发布原理图和 PCB 设计。
- 分级管理：不同项目类型采用不同流程深度。
- 门禁受控：阶段准出必须有评审结论、问题清单和责任人。
- 允许并行：在风险可接受且有记录的情况下，允许部分活动并行推进。
- 问题闭环：关键问题必须有责任人、计划完成时间和验证结论。
- 变更受控：需求、设计、BOM、工艺和测试变更必须按变更流程管理。
- 资料归档：每阶段输出文件必须按版本、状态和目录要求归档。

## 3. IPD 流程总体框架

### 3.1 阶段模型

```mermaid
flowchart LR
    A["概念阶段<br/>机会识别与立项判断"] --> B["计划阶段<br/>需求、方案、计划与风险"]
    B --> C["开发阶段<br/>设计、打样、调试与设计评审"]
    C --> D["验证阶段<br/>EVT / DVT / 可靠性 / 认证适用性验证"]
    D --> E["发布阶段<br/>PVT、试产、量产发布与归档"]
    E --> F["生命周期阶段<br/>变更、质量、供应与退市管理"]
```

### 3.2 阶段门禁

| 阶段 | 关键门禁 | 通过后允许进入 |
|---|---|---|
| 概念阶段 | 立项 / 概念评审 | 计划阶段 |
| 计划阶段 | 产品需求评审、硬件方案评审 | 开发阶段 |
| 开发阶段 | 原理图评审、PCB 评审、样机调试评审 | 验证阶段 |
| 验证阶段 | EVT 评审、DVT 评审、专项测试评审 | 发布阶段 |
| 发布阶段 | PVT 评审、量产发布评审 | 生命周期阶段 |
| 生命周期阶段 | 工程变更评审、质量问题评审、器件替代评审、退市评审 | 持续维护或退市 |

### 3.3 并行开发与例外放行

实际项目中，结构、固件、采购、样机准备和测试准备可在满足条件时并行推进。

| 并行动作 | 允许条件 | 风险控制 |
|---|---|---|
| 方案评审前启动关键器件样品申请 | 关键器件方向基本确定，采购周期影响项目进度 | 记录为预采购，不作为量产选型冻结 |
| 原理图评审前进行结构预布局 | 接口位置和板框方向基本明确 | 标记为预设计，正式投板前必须复核 |
| PCB 完成前启动测试方案草案 | 需求和接口基本稳定 | 测试用例随设计变更同步更新 |
| EVT 问题未完全关闭进入 DVT 准备 | 未关闭问题不影响 DVT 准备活动 | 不得执行正式 DVT 准出评审 |

例外放行必须形成《阶段例外放行记录》，至少包含：

- 例外事项。
- 未完成内容。
- 影响范围。
- 风险等级。
- 责任人。
- 计划关闭时间。
- 批准人。
- 回退条件。

## 4. 项目分级与流程裁剪规则

### 4.1 项目等级定义

| 项目等级 | 适用项目 | 流程要求 |
|---|---|---|
| A 类 | 全新产品、平台型产品、重大客户定制、涉及新技术或新供应链的项目 | 执行完整 IPD 六阶段 |
| B 类 | 派生产品、普通客户定制、局部功能扩展、已有平台上的硬件改型 | 执行简化 IPD 流程，可合并概念与计划阶段 |
| C 类 | 降本优化、器件替代、质量改进、小范围设计优化 | 执行变更驱动流程，以评估、验证、发布为主 |
| R 类 | 预研项目、技术探索、方案验证 | 执行预研流程，可不进入发布和量产阶段 |

### 4.2 项目类型裁剪表

| 项目类型 | 默认等级 | 必须执行 | 可裁剪 / 合并 | 不可裁剪 |
|---|---|---|---|---|
| 全新产品开发 | A 类 | 六阶段完整流程 | 无 | 需求、方案、设计评审、验证、发布、归档 |
| 平台型产品开发 | A 类 | 六阶段完整流程 | 可复用历史可行性资料 | 差异需求、平台风险、验证、发布 |
| 派生产品开发 | B 类 | 计划、开发、验证、发布 | 概念与计划可合并 | 差异需求、BOM、回归验证、归档 |
| 客户定制项目 | B 类 | 需求确认、设计、验证、交付评审 | 概念与计划可合并 | 客户需求确认、验收标准、版本追溯 |
| 降本优化项目 | C 类 | ECR、影响分析、验证、ECN | MRD / PRD 可不单独编制 | 成本目标、风险评估、验证记录 |
| 器件替代项目 | C 类 | 替代评估、样机验证、BOM 变更、ECN | 概念、计划阶段可不执行 | 替代风险、供应风险、回归验证 |
| 质量改进项目 | C 类 | 问题分析、措施验证、变更发布 | 商业分析可不执行 | 根因分析、验证结论、闭环记录 |
| 预研项目 | R 类 | 技术目标、实验方案、实验记录、结论评审 | 量产发布阶段可不执行 | 技术结论、风险结论、资料归档 |

### 4.3 裁剪审批要求

流程裁剪必须在项目启动时确定，并在《项目计划》或《项目裁剪说明》中记录。裁剪内容不得影响安全、法规、客户验收、量产质量和设计追溯。

## 5. IPD 分阶段导入路径

由于企业当前无正式 IPD 流程，建议采用渐进式导入。

### 5.1 第一阶段：最小必需流程

适用时间：导入初期 1-3 个月。

必须执行：

- 项目等级判定。
- 产品需求评审。
- 硬件方案评审。
- 原理图评审。
- PCB 评审。
- 问题跟踪表。
- 样机调试报告。
- 量产发布或变更发布评审。
- 设计归档清单。

### 5.2 第二阶段：标准流程固化

适用时间：导入 3-6 个月。

增加执行：

- 阶段输入输出矩阵。
- RACI 职责矩阵。
- EVT / DVT / PVT 阶段定义。
- 需求追溯矩阵。
- BOM 评审。
- 工程变更流程。
- 项目复盘。

### 5.3 第三阶段：流程度量与优化

适用时间：导入 6 个月后。

增加执行：

- 阶段准出质量评分。
- 设计返板次数统计。
- 问题关闭周期统计。
- 试产良率统计。
- 量产质量问题复盘。
- 流程稽核和模板优化。

## 6. 组织职责与角色兼任

### 6.1 组织机制

建议建立轻量化 IPD / PDT 机制，而不是一次性搭建复杂组织。

| 角色 | 主要职责 | 小团队兼任建议 |
|---|---|---|
| PDT 负责人 | 项目目标、阶段门禁、重大风险和资源协调 | 可由研发负责人、产品负责人或项目负责人兼任 |
| 项目经理 | 计划、会议、问题闭环、资料归档推动 | 可由项目负责人或产品经理兼任 |
| 产品经理 | 客户需求、PRD、验收标准、需求变更 | 不建议由硬件负责人长期兼任 |
| 系统工程师 | 系统方案、跨模块接口、需求分解 | 可由资深硬件或固件负责人兼任 |
| 硬件负责人 | 硬件需求、方案、设计、调试、验证 | 必设 |
| PCB 工程师 | PCB Layout、工艺约束、投板资料 | 可由硬件工程师兼任或外协 |
| 固件负责人 | 固件方案、接口、联调、版本配合 | 必设 |
| 测试负责人 | 测试计划、测试报告、验证结论 | 可由测试工程师或质量工程师兼任 |
| 质量负责人 | 质量目标、问题闭环、发布准入 | 可由质量部门指定接口人 |
| 工艺 / 生产负责人 | DFM、PVT、生产测试、工艺文件 | 可按阶段参与 |
| 采购 / 供应链负责人 | 器件供应、成本、交期、替代料 | 可按关键节点参与 |

### 6.2 最低参会要求

| 评审 | 必须参加 | 可选参加 |
|---|---|---|
| 产品需求评审 | 产品、项目、硬件、固件、测试 | 质量、结构、销售 |
| 硬件方案评审 | 项目、硬件、固件、测试、采购 | 结构、质量、生产 |
| 原理图评审 | 硬件、固件、测试、同行评审人 | 质量、生产 |
| PCB 评审 | 硬件、PCB、结构、工艺 / 生产 | 测试、EMC、质量 |
| EVT / DVT 评审 | 项目、硬件、固件、测试、质量 | 产品、生产 |
| PVT / 量产发布评审 | 项目、硬件、生产、工艺、质量、供应链 | 产品、售后 |
| 工程变更评审 | 变更提出人、硬件、质量、生产 / 采购 | 产品、售后、项目 |

## 7. 阶段管理通用要求

每个阶段均应定义：

- 阶段目标。
- 启动条件。
- 必需输入。
- 主要活动。
- 必需输出。
- 滚动更新文件。
- 评审点。
- 责任人。
- 准出标准。
- 可并行事项。
- 可裁剪事项。
- 风险与问题闭环。
- 变更控制。
- 归档要求。

### 7.1 文件类型分级

| 文件类型 | 定义 | 管理要求 |
|---|---|---|
| 冻结类文件 | 进入下一阶段的设计和决策基线 | 阶段评审后受控，修改需走变更 |
| 滚动类文件 | 随项目推进持续更新的文件 | 每次评审确认当前状态 |
| 记录类文件 | 会议、测试、问题、调试等过程记录 | 真实完整即可，不追求过度美化 |
| 模板类文件 | 用于统一格式和检查项 | 可按项目复杂度裁剪 |

## 8. 概念阶段规范

### 8.1 阶段目标

明确市场机会、客户需求、产品定位、技术可行性、初步成本、初步风险和项目价值，决定是否立项。

### 8.2 适用范围

完整概念阶段主要适用于 A 类项目。B 类项目可与计划阶段合并，C 类项目可不单独执行概念阶段。

### 8.3 主要活动

- 收集市场、客户、竞品和应用场景信息。
- 明确教育教学、实验管理、考试竞赛和区域管理等应用场景。
- 分析产品机会、客户价值和商业可行性。
- 识别关键技术难点、供应风险、认证风险和项目资源需求。
- 初步判断项目等级和流程深度。

### 8.4 输出文件

| 文件 | 类型 | A 类 | B 类 | C 类 / R 类 |
|---|---|---|---|---|
| 《市场需求说明书 MRD》 | 冻结类 | 必需 | 可合并到 PRD | 可不适用 |
| 《产品机会分析报告》 | 记录类 | 必需 | 可简化 | 可不适用 |
| 《技术可行性分析报告》 | 冻结类 | 必需 | 按风险决定 | R 类必需 |
| 《初步风险清单》 | 滚动类 | 必需 | 必需 | 必需 |
| 《项目等级与流程裁剪说明》 | 冻结类 | 必需 | 必需 | 必需 |
| 《概念阶段评审报告》 | 记录类 | 必需 | 可合并 | 可不适用 |

## 9. 计划阶段规范

### 9.1 阶段目标

将市场需求转化为可设计、可验证、可交付的产品需求和硬件方案，形成项目计划、资源计划、成本目标、验证策略和风险计划。

### 9.2 主要活动

- 编制和评审 PRD。
- 分解硬件需求和接口需求。
- 建立需求追溯矩阵。
- 完成硬件总体方案、关键器件选型和技术路线。
- 明确样机、测试、认证适用性、工艺和量产导入计划。
- 建立风险清单、问题跟踪机制和版本管理方式。

### 9.3 输出文件

| 文件 | 类型 | 管理要求 |
|---|---|---|
| 《产品需求说明书 PRD》 | 冻结类 | 需求基线，变更需记录 |
| 《需求追溯矩阵 TRM》 | 滚动类 | 贯穿需求、设计、测试和验收 |
| 《项目任务书》 | 冻结类 | 明确范围、目标、资源和里程碑 |
| 《项目计划》 | 滚动类 | 可按阶段滚动更新 |
| 《硬件需求规格书》 | 冻结类 | 必须追溯到 PRD |
| 《硬件总体方案设计说明书》 | 冻结类 | 方案评审通过后受控 |
| 《关键器件选型报告》 | 冻结 + 滚动 | 关键器件冻结，替代料可滚动补充 |
| 《初版 BOM / 成本估算》 | 滚动类 | 计划阶段形成初版，不要求完全冻结 |
| 《测试策略 / 验证计划》 | 滚动类 | 先形成覆盖策略，详细用例可后续完善 |
| 《法规与认证适用性判定表》 | 冻结类 | 明确是否涉及 EMC、安规、环保、无线等要求 |
| 《风险管理计划》 | 滚动类 | 每次阶段评审更新 |
| 《计划阶段评审报告》 | 记录类 | 准出必需 |

## 10. 开发阶段规范

### 10.1 阶段目标

完成硬件详细设计、原理图、PCB、BOM、样机制作和单板调试，确保设计具备进入系统验证的条件。

### 10.2 主要活动

- 完成硬件详细设计。
- 完成关键器件确认和替代料策略。
- 完成原理图设计与评审。
- 完成 PCB Layout、叠层、布线、基础 EMC、基础 DFM、基础 DFT 检查。
- 完成 BOM 编制、物料状态确认和成本更新。
- 完成样机制作、焊接、调试和问题记录。
- 更新需求追溯矩阵。

### 10.3 输出文件

| 文件 | 类型 | 管理要求 |
|---|---|---|
| 《硬件详细设计说明书》 | 冻结类 | 可按模块分册编制 |
| 原理图文件 | 冻结类 | 原理图评审后受控 |
| 《原理图设计检查表》 | 记录类 | 必需 |
| PCB Layout 文件 | 冻结类 | PCB 评审后受控 |
| 《PCB Layout 检查表》 | 记录类 | 必需 |
| 《BOM 清单》 | 滚动 + 冻结 | 样机 BOM、试产 BOM、量产 BOM 分阶段冻结 |
| 《器件承认资料》 | 滚动类 | 关键器件优先完成 |
| 《基础 DFM 检查表》 | 记录类 | 样机投板前完成 |
| 《基础 DFT 检查表》 | 记录类 | 样机投板前完成 |
| 《样机制作申请》 | 记录类 | 按样机版本管理 |
| 《样机调试报告》 | 记录类 | 进入 EVT 前必须完成 |
| 《问题跟踪表》 | 滚动类 | 持续维护 |
| 《开发阶段评审报告》 | 记录类 | 准出必需 |

## 11. 验证阶段规范

### 11.1 阶段目标

通过 EVT、DVT、可靠性、专项测试和认证适用性验证，确认产品设计满足需求、质量目标和量产导入要求。

### 11.2 主要活动

- 执行 EVT，验证工程样机基本功能、性能、接口和调试完整性。
- 执行 DVT，验证设计完整性、稳定性、环境适应性和可靠性。
- 根据《法规与认证适用性判定表》执行 EMC、安规、环保、无线、行业认证等预检或正式测试。
- 闭环设计问题、测试问题和供应问题。
- 更新设计文件、BOM、测试文件、风险清单和需求追溯矩阵。

### 11.3 输出文件

| 文件 | 类型 | 管理要求 |
|---|---|---|
| 《EVT 测试计划》 | 滚动类 | EVT 前完成 |
| 《EVT 测试报告》 | 记录类 | EVT 评审必需 |
| 《DVT 测试计划》 | 滚动类 | DVT 前完成 |
| 《DVT 测试报告》 | 记录类 | DVT 评审必需 |
| 《可靠性测试报告》 | 记录类 | 按产品等级和风险执行 |
| 《EMC 测试报告 / 预检报告》 | 记录类 | 按适用性判定执行 |
| 《安规测试报告 / 判定记录》 | 记录类 | 按适用性判定执行 |
| 《需求追溯矩阵 TRM》 | 滚动类 | 记录需求验证覆盖情况 |
| 《问题跟踪表》 | 滚动类 | S1 / S2 问题必须重点闭环 |
| 《设计变更记录》 | 记录类 | 与问题闭环关联 |
| 《验证阶段评审报告》 | 记录类 | 准出必需 |

## 12. 发布阶段规范

### 12.1 阶段目标

完成 PVT、试产验证、生产测试、工艺确认、量产 BOM、完整 DFM / DFT 和设计资料归档，确认产品具备稳定量产条件。

### 12.2 主要活动

- 制定并执行 PVT 试产计划。
- 验证生产工艺、测试方案、工装夹具和生产效率。
- 完成完整 DFM / DFT / DFA 检查。
- 确认量产 BOM、供应链状态和关键物料交期。
- 完成生产导入检查和量产发布评审。
- 完成设计归档。

### 12.3 量化准出指标

正式规范应结合企业实际设定具体阈值。V1.1 建议先采用可调整指标。

| 指标 | 建议准出要求 |
|---|---|
| PVT 试产数量 | 按产品复杂度设定，原则上能覆盖主要生产波动 |
| 试产直通率 / 良率 | 达到项目设定目标，未达标需有批准和改善计划 |
| 关键不良项 | S1 为 0，S2 原则上关闭 |
| 生产测试覆盖率 | 覆盖关键功能、关键接口、关键安全项 |
| 测试节拍 | 满足计划产能要求或有改善计划 |
| 量产 BOM | 受控发布，关键物料供应风险可接受 |
| 工艺文件齐套率 | 100% 齐套后方可量产发布 |
| 设计归档齐套率 | P0 文件 100% 齐套 |

### 12.4 输出文件

| 文件 | 类型 | 管理要求 |
|---|---|---|
| 《PVT 试产计划》 | 冻结类 | PVT 前完成 |
| 《PVT 试产报告》 | 记录类 | PVT 评审必需 |
| 《生产测试方案》 | 冻结类 | 量产发布前受控 |
| 《工艺文件》 | 冻结类 | 量产发布前齐套 |
| 《完整 DFM / DFT / DFA 检查表》 | 记录类 | PVT 或量产发布前完成 |
| 《量产 BOM》 | 冻结类 | 必须受控发布 |
| 《生产导入检查表》 | 记录类 | 量产发布必需 |
| 《量产发布评审报告》 | 记录类 | 准出必需 |
| 《设计归档清单》 | 记录类 | P0 文件必须齐套 |

## 13. 生命周期阶段规范

### 13.1 阶段目标

在量产后持续管理质量、成本、供应、变更、客户问题和退市活动，保障产品全生命周期受控。

### 13.2 子流程划分

生命周期阶段不作为单一大流程执行，而拆分为以下子流程：

| 子流程 | 触发条件 | 主要输出 |
|---|---|---|
| 工程变更流程 | 需求、设计、BOM、工艺、测试或供应变化 | ECR、ECO、ECN |
| 量产质量问题闭环流程 | 生产、客户、售后反馈质量问题 | 质量问题分析报告、纠正预防措施 |
| 器件 PCN / EOL 管理流程 | 供应商通知变更、停产或替代 | 器件替代评估、验证报告、ECN |
| 降本优化流程 | 成本目标或供应策略调整 | 降本方案、验证记录、BOM 变更 |
| 产品退市流程 | 产品停产、市场退出或替代产品发布 | 停产 / 退市评估、备件计划、客户通知 |

### 13.3 输出文件

- 《工程变更申请 ECR》。
- 《工程变更执行 ECO》。
- 《工程变更通知 ECN》。
- 《量产问题分析报告》。
- 《质量改进报告》。
- 《器件替代评估报告》。
- 《器件 PCN / EOL 处理记录》。
- 《产品停产 / 退市评估》。

## 14. 需求追溯管理规范

### 14.1 追溯目标

确保产品需求、硬件需求、设计实现、测试用例和测试结论之间可追溯，避免“需求写了但没设计、设计了但没测试、测试了但不知道覆盖什么”。

### 14.2 追溯关系

```mermaid
flowchart LR
    A["PRD 产品需求"] --> B["硬件需求规格"]
    B --> C["硬件方案 / 详细设计"]
    C --> D["测试用例"]
    D --> E["测试结果"]
    E --> F["验收结论"]
```

### 14.3 最小字段

《需求追溯矩阵》至少包含：

- 需求编号。
- 需求描述。
- 来源。
- 优先级。
- 对应硬件需求。
- 对应设计章节 / 电路模块。
- 对应测试用例。
- 测试阶段。
- 测试结论。
- 问题编号。
- 当前状态。

## 15. 法规与认证适用性判定规范

### 15.1 判定目标

在计划阶段识别产品是否涉及 EMC、安规、环保、无线、电池、儿童接触、海外销售和行业认证要求，避免验证阶段才发现认证缺口。

### 15.2 判定维度

| 维度 | 示例问题 |
|---|---|
| 销售区域 | 是否国内销售、海外销售或多区域销售 |
| 供电方式 | 是否市电供电、外置适配器供电、电池供电 |
| 通信方式 | 是否含无线通信、蓝牙、Wi-Fi、蜂窝通信 |
| 使用人群 | 是否面向儿童、学生或公共教学场景 |
| 使用环境 | 是否实验室、教室、户外、潮湿、高温等场景 |
| 接触方式 | 是否可直接接触金属端子、传感器、电极或运动部件 |
| 材料要求 | 是否涉及环保、阻燃、有害物质限制 |
| 行业要求 | 是否涉及教育装备、考试设备、竞赛设备等特殊要求 |

### 15.3 输出要求

计划阶段必须形成《法规与认证适用性判定表》。如判定适用，应在验证计划中明确测试项目、责任人、样机版本、送测时间和准出标准。

## 17. 文件、版本与归档规范

### 17.1 文件编号规则

建议采用“文档编号 + 项目代号 + 文件类型 + 修订版”的方式，日期和状态作为文档属性管理。

## 19. 流程执行与持续改进

### 19.1 执行保障

为避免制度发布后执行弱，应建立以下机制：

- 新项目启动时进行流程裁剪确认。
- 阶段评审前由项目经理进行资料预检查。
- 阶段评审结论必须进入问题跟踪表。
- 每月抽查项目资料归档情况。
- 每个项目结束后进行项目复盘。

### 19.2 度量指标

建议跟踪以下指标：

| 指标 | 目的 |
|---|---|
| 需求变更次数 | 评估需求质量和需求冻结效果 |
| PCB 返板次数 | 评估硬件设计质量 |
| EVT / DVT 问题数量 | 评估设计成熟度 |
| S1 / S2 问题关闭周期 | 评估问题闭环效率 |
| PVT 试产良率 | 评估量产导入质量 |
| 量产前三个月质量问题数量 | 评估发布质量 |
| 归档齐套率 | 评估流程执行质量 |
| 工程变更及时率 | 评估变更管理有效性 |

### 19.3 复盘机制

每个 A 类项目和重大 B 类项目在量产发布后应进行复盘，至少输出：

- 项目目标达成情况。
- 需求变更分析。
- 设计问题分析。
- 返板原因分析。
- 试产问题分析。
- 量产初期质量问题分析。
- 流程执行问题。
- 后续改进措施。

## 21. 下一步建议

建议先确认以下内容：

- 项目等级 A / B / C / R 是否符合企业实际。
