# 硬件研发设计规范 V1.0 大纲

| 项目 | 内容 |
|---|---|
| 文档名称 | 硬件研发设计规范 |
| 版本 | V1.0 草案 |
| 适用企业 | 江苏苏威尔科技有限公司 |
| 适用范围 | 教育装备类硬件产品研发、验证、导入、量产维护与工程变更 |
| 流程框架 | IPD 六阶段：概念、计划、开发、验证、发布、生命周期 |
| 文档定位 | 企业制度文件 + 项目执行指南 + 模板索引 |

## 1. 总则

### 1.1 目的

建立统一、可执行、可评审、可追溯的硬件研发设计规范，解决研发流程不完整、执行弱、评审缺失、文档不齐、问题闭环不足和跨部门协同不稳定等问题。

### 1.2 适用范围

适用于公司教育装备类产品的硬件研发活动，包括：

- 全新产品开发。
- 平台型产品开发。
- 派生产品开发。
- 客户定制项目。
- 降本优化项目。
- 器件替代项目。
- 质量改进项目。
- 预研项目。

### 1.3 适用对象

适用于产品、项目、系统、硬件、PCB、固件、结构、测试、可靠性、EMC、工艺、生产、采购、供应链、质量、售后和 IPD / PDT 相关人员。

### 1.4 基本原则

- 需求先行：无确认需求，不进入详细设计。
- 方案先行：无总体方案评审，不启动原理图和 PCB 正式设计。
- 评审门禁：阶段输出未达标，不得进入下一阶段。
- 问题闭环：所有关键问题必须有责任人、计划完成时间和验证结论。
- 变更受控：需求、设计、BOM、工艺和测试变更必须纳入 ECR / ECO / ECN 管理。
- 资料归档：每阶段输出文件必须按版本、状态和目录要求归档。

## 3. IPD 流程总体框架

### 3.1 阶段模型

```mermaid
flowchart LR
    A["概念阶段<br/>机会识别与可行性"] --> B["计划阶段<br/>需求、方案与计划"]
    B --> C["开发阶段<br/>设计、打样与调试"]
    C --> D["验证阶段<br/>EVT / DVT / 认证验证"]
    D --> E["发布阶段<br/>PVT、试产与量产发布"]
    E --> F["生命周期阶段<br/>维护、变更与退市"]
```

### 3.2 阶段门禁

| 阶段 | 关键门禁 | 通过后允许进入 |
|---|---|---|
| 概念阶段 | 立项 / 概念评审 | 计划阶段 |
| 计划阶段 | 产品需求与硬件方案评审 | 开发阶段 |
| 开发阶段 | 设计发布与样机调试评审 | 验证阶段 |
| 验证阶段 | EVT / DVT / 可靠性 / EMC 评审 | 发布阶段 |
| 发布阶段 | PVT 与量产发布评审 | 生命周期阶段 |
| 生命周期阶段 | 工程变更评审、质量改进评审、退市评审 | 持续维护或退市 |

### 3.3 项目分类与流程深度

当前 V1.0 草案按“完整流程”为默认要求。后续可按项目等级增加裁剪规则。

| 项目类型 | V1.0 默认流程 | 后续可裁剪方向 |
|---|---|---|
| 全新产品开发 | 完整六阶段 | 不建议裁剪 |
| 平台型产品开发 | 完整六阶段 | 可复用平台资料 |
| 派生产品开发 | 完整六阶段 | 可简化概念阶段 |
| 客户定制项目 | 完整六阶段 | 可合并概念与计划评审 |
| 降本优化项目 | 完整六阶段 | 可聚焦变更、验证和发布 |
| 器件替代项目 | 完整六阶段 | 可简化为变更评审 + 回归验证 |
| 质量改进项目 | 完整六阶段 | 可按问题闭环流程执行 |
| 预研项目 | 完整六阶段 | 可不进入发布阶段 |

## 4. 组织职责

### 4.1 组织机制

建议建立 IPD / PDT 项目组织机制：

- IPD 决策层：负责项目立项、重大资源、阶段门禁和量产发布批准。
- PDT 负责人：负责跨部门项目目标、计划、风险和交付。
- 项目经理：负责项目计划、进度、会议、问题闭环和资料归档推动。
- 硬件负责人：负责硬件需求、方案、设计、调试、验证和硬件归档。
- 测试负责人：负责测试策略、测试计划、测试执行和测试报告。
- 质量负责人：负责质量目标、评审合规性、问题闭环和量产质量准入。
- 工艺 / 生产负责人：负责可制造性、试产导入、生产测试和工艺资料。
- 采购 / 供应链负责人：负责关键器件、供应风险、成本和交期。

### 4.2 RACI 职责矩阵

正式规范中应包含 RACI 矩阵，至少覆盖以下活动：

- 需求确认。
- 立项评审。
- 硬件需求分析。
- 硬件方案设计。
- 关键器件选型。
- 原理图设计与评审。
- PCB Layout 与评审。
- BOM 发布。
- 样机制作与调试。
- EVT / DVT / PVT。
- 量产发布。
- 工程变更。
- 设计归档。

## 6. 概念阶段规范

### 6.1 阶段目标

明确市场机会、客户需求、产品定位、技术可行性、初步成本、初步风险和项目价值，决定是否立项。

### 6.2 主要活动

- 收集市场、客户、竞品和应用场景信息。
- 明确教育教学、实验管理、考试竞赛和区域管理等应用场景。
- 分析产品机会、客户价值和商业可行性。
- 识别关键技术难点、供应风险、认证风险和项目资源需求。
- 输出初步立项建议。

### 6.3 输出文件

- 《市场需求说明书 MRD》。
- 《产品机会分析报告》。
- 《技术可行性分析报告》。
- 《初步风险清单》。
- 《初步成本估算》。
- 《概念阶段评审报告》。

## 7. 计划阶段规范

### 7.1 阶段目标

将市场需求转化为可设计、可验证、可交付的产品需求和硬件方案，形成项目计划、资源计划、成本目标和验证策略。

### 7.2 主要活动

- 编制和评审 PRD。
- 分解硬件需求和接口需求。
- 完成硬件总体方案、关键器件选型和技术路线。
- 明确样机、测试、认证、工艺和量产导入计划。
- 建立风险清单、问题跟踪机制和版本管理方式。

### 7.3 输出文件

- 《产品需求说明书 PRD》。
- 《项目任务书》。
- 《项目计划》。
- 《硬件需求规格书》。
- 《硬件总体方案设计说明书》。
- 《关键器件选型报告》。
- 《初版 BOM / 成本估算》。
- 《测试策略 / 验证计划》。
- 《风险管理计划》。
- 《计划阶段评审报告》。

## 8. 开发阶段规范

### 8.1 阶段目标

完成硬件详细设计、原理图、PCB、BOM、样机制作和单板调试，确保设计具备进入系统验证的条件。

### 8.2 主要活动

- 完成硬件详细设计。
- 完成关键器件确认和替代料策略。
- 完成原理图设计与评审。
- 完成 PCB Layout、叠层、布线、EMC、DFM、DFT 检查。
- 完成 BOM 编制、物料状态确认和成本更新。
- 完成样机制作、焊接、调试和问题记录。

### 8.3 输出文件

- 《硬件详细设计说明书》。
- 原理图文件。
- 《原理图设计检查表》。
- PCB Layout 文件。
- 《PCB Layout 检查表》。
- 《BOM 清单》。
- 《器件承认资料》。
- 《DFM 检查表》。
- 《DFT 检查表》。
- 《样机制作申请》。
- 《样机调试报告》。
- 《开发阶段评审报告》。

## 9. 验证阶段规范

### 9.1 阶段目标

通过 EVT、DVT、可靠性、EMC、安规和系统级验证，确认产品设计满足需求、质量目标和量产导入要求。

### 9.2 主要活动

- 执行 EVT，验证工程样机的基本功能、性能和接口。
- 执行 DVT，验证设计完整性、稳定性、环境适应性和可靠性。
- 执行 EMC、安规、环保、行业认证等预检或正式测试。
- 闭环设计问题、测试问题和供应问题。
- 更新设计文件、BOM、测试文件和风险清单。

### 9.3 输出文件

- 《EVT 测试计划》。
- 《EVT 测试报告》。
- 《DVT 测试计划》。
- 《DVT 测试报告》。
- 《EMC 测试报告》。
- 《安规测试报告》。
- 《可靠性测试报告》。
- 《问题跟踪表》。
- 《设计变更记录》。
- 《验证阶段评审报告》。

## 10. 发布阶段规范

### 10.1 阶段目标

完成 PVT、试产验证、生产测试、工艺确认、量产 BOM 和设计资料归档，确认产品可稳定量产。

### 10.2 主要活动

- 制定并执行 PVT 试产计划。
- 验证生产工艺、测试方案、工装夹具和生产效率。
- 确认量产 BOM、供应链状态和关键物料交期。
- 完成生产导入检查和量产发布评审。
- 完成设计归档。

### 10.3 输出文件

- 《PVT 试产计划》。
- 《PVT 试产报告》。
- 《生产测试方案》。
- 《工艺文件》。
- 《量产 BOM》。
- 《生产导入检查表》。
- 《量产发布评审报告》。
- 《设计归档清单》。

## 11. 生命周期阶段规范

### 11.1 阶段目标

在量产后持续管理质量、成本、供应、变更、客户问题和退市活动，保障产品全生命周期受控。

### 11.2 主要活动

- 收集量产问题、售后问题和客户反馈。
- 执行质量问题分析和改进。
- 管理器件替代、BOM 变更、设计变更和工艺变更。
- 评估停产、退市和备件策略。

### 11.3 输出文件

- 《工程变更申请 ECR》。
- 《工程变更执行 ECO》。
- 《工程变更通知 ECN》。
- 《量产问题分析报告》。
- 《质量改进报告》。
- 《器件替代评估报告》。
- 《产品停产 / 退市评估》。

## 13. 文件、版本与归档规范

### 13.1 文件命名规则

建议格式：

`项目代号_产品名称_文件类型_版本号_日期_状态`

示例：

`SWL-P001_智能实验设备_硬件总体方案_V0.3_2026-06-29_评审稿.md`

### 13.2 版本状态

| 状态 | 含义 |
|---|---|
| 草稿 | 编制中，未正式评审 |
| 评审稿 | 已提交评审 |
| 发布稿 | 已通过评审，可执行 |
| 归档稿 | 项目阶段完成后归档 |
| 作废 | 已被新版本替代 |

### 13.3 归档目录结构

建议项目归档目录：

```text
项目名称/
  00_项目管理/
  01_需求与立项/
  02_方案与计划/
  03_硬件设计/
  04_PCB与结构协同/
  05_BOM与供应链/
  06_样机与调试/
  07_测试验证/
  08_试产与量产/
  09_工程变更/
  10_评审记录/
  11_归档发布/
```

## 16. 下一步建议

建议先确认本大纲的以下内容：

- IPD 六阶段是否符合企业管理习惯。
- 项目类型是否全部坚持完整流程。
- 是否需要补充“认证阶段”或“客户定制阶段”。
- 是否确认采用 RACI 职责矩阵。
- 技术规范章节是否需要加深到具体电路设计约束。
