## 2. 本版依据

本版基于已填写的沟通文档形成，核心依据如下：

- 企业：江苏苏威尔科技有限公司。
- 行业：教育装备。
- 产品应用：K12 及师范院校理科实验教学、实验管理、标准化考试、大规模竞赛和区域教育管理。
- 客户类型：国内外 K-12 学校、教育管理部门、商业合作伙伴。
- 产品复杂度：中等复杂度。
- 团队规模：研发约 50 人，硬件团队约 7 人，固件团队约 6 人。
- 当前流程状态：已有研发流程但不完整，且执行偏弱。
- IPD 状态：当前无正式 IPD 流程。
- 发布目标：作为企业制度文件发布。
- 阶段要求：采用概念、计划、开发、验证、发布、生命周期六阶段。
- 阶段管理要求：每阶段均需要目标、启动条件、输入、活动、输出、评审、检查表、责任人、准出、风险、问题闭环、变更控制和归档要求。
- 管理要求：需要文件命名规则、版本管理规则和归档目录结构。

## 4. 可更改规则

后续修改建议采用“变更请求”方式，不直接覆盖已确认文件。

### 4.1 小改动

适用场景：

- 调整术语。
- 修改角色名称。
- 增加或删除一个输出文件。
- 修改某个阶段准出条件。

建议版本号：

- `v1.0.1`
- `v1.0.2`

### 4.2 中等改动

适用场景：

- 调整 IPD 阶段划分。
- 增加项目裁剪规则。
- 重构评审体系。
- 增加新的模板包。

建议版本号：

- `v1.1`
- `v1.2`

### 4.3 大改动

适用场景：

- 企业组织架构变化。
- 产品线范围显著扩大。
- 从草案转为正式发布版。
- 需要输出 Word、Excel、流程图和模板全集。

建议版本号：

- `v2.0`

## 5. 可回退规则

当前目录不是 git 仓库，因此本阶段不依赖 git 回退。回退采用文件包基线方式。

### 5.1 回退到原始输入

如需重新生成，可基于以下快照恢复：

`_source_snapshot/hardware-rd-design-spec-communication.filled.2026-06-29.md`

### 5.2 回退到某个版本包

每次大版本或中版本修改，建议新建独立目录，例如：

- `outputs/2026-06-29_v1.0-draft`
- `outputs/2026-06-29_v1.1-draft`
- `outputs/2026-06-29_v2.0-release`

回退时，直接使用上一版本目录作为基线即可。

### 5.3 单文件回退

如只需回退某一文件：

1. 找到上一版本目录中的同名文件。
2. 对比差异。
3. 将上一版文件复制为当前修改基线。
4. 在本文件的“变更记录”中记录原因。
